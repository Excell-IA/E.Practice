## Contesto

Ripartenza dopo circa dieci giorni di pausa. Obiettivo operativo: rimettere E.Practice
sul percorso prodotto, abbandonando la demo isolata e portando il modulo verso il
database condiviso E.Work (`ework`) e verso E.Contacts come rubrica autorevole.

La timeline operativa e' su Airtable. Il task coinvolto oggi e':

- PR155 - migrazione E.Practice verso storage SQL condiviso E.Work.

## Decisione architetturale emersa oggi

La prima implementazione aveva creato una migration nel repo core E.Work con le
tabelle `practice_*`. Questa scelta ha fatto emergere un dubbio corretto:
mettere DDL dei moduli nel core E.Work crea collisioni tra agenti e tra migration
parallele.

Decisione piattaforma comunicata dal workspace E.Work:

- il core E.Work possiede solo tabelle condivise, ad esempio `tenants`, `users`,
  `audit`, `module_status`;
- le tabelle di ogni modulo vivono nel repo del modulo;
- E.Practice deve quindi creare le proprie tabelle `practice_*` nel repo
  E.Practice, con Alembic branch label dedicata `practice`;
- il modulo si collega comunque al database condiviso `ework`;
- il branch DDL creato nel repo E.Work non deve essere mergiato.

Conseguenza pratica:

- adapter SQL lato E.Practice: corretto, si tiene;
- DDL nel repo E.Work: abbandonato come approccio, usato solo come riferimento
  tecnico temporaneo;
- prossima DDL ufficiale: migration Alembic nel repo E.Practice.
